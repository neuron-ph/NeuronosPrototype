## Context

Marcus dumped a 7-image + text batch (2026-07-18) reworking the whole E-Voucher / accounting-movement flow. The through-line: **every accounting movement should pass through a pre-posting "Transaction Journal" before it hits the General Journal**, each line linked to a real COA account and (for expenses) a real booking. Today the accounting side is fragmented — posting is scattered across per-document "GL Posting Sheets" and disbursement code, invoices/collections write empty-lines placeholder JEs, the catalog and COA are disjoint, and booking linkage is one-per-voucher. This batch tightens all of it onto one spine.

18 tickets were opened + refined conversationally, plus this plan. Nothing is built yet — all `Todo`, awaiting **Go Ahead**. Dev-first always; **no prod** without explicit per-turn permission.

## Architecture decisions (locked with Marcus)

1. **~~Transaction Journal = separate table~~ → REVERSED (Marcus 7/18): ONE journal, two lenses.** The TJ and GJ are the **same table (`journal_entries`) and same component (`GeneralJournal`)**. The GJ is just the **posted-only view**; the TJ is the full pipeline (all statuses). "Posting" is a **status flip** (`ready_to_post → posted`), not a cross-table copy. Pre-posting stages (`pending`/`awaiting_ack`/`ready_to_post`) added to `journal_entries.status`; lifecycle columns added (migration 244, which drops the separate table from 243). Balances still read `status='posted'`, so pipeline rows are inert until posted. *(NEU-099)*
2. **TJ is the UNIVERSAL pre-posting layer** — expense, cash-advance, invoice/AR, collection, transfer all flow through it. It **consolidates** today's scattered posting (DisburseEVoucherPage, InvoiceGLPostingSheet, CollectionGLPostingSheet, empty-lines placeholders) into one seam. *(NEU-099, NEU-106)*
3. **Catalog→COA link is an editable autofill**, not a hard binding — the account seeds each TJ line but can be changed at journal time. *(NEU-091)*
4. **Cash Advance = allocation plan (Model A)** — CA line items = booking + amount, reuse `evoucher_line_items` + per-line booking, no new budget tables, no ceiling. *(NEU-104)*
5. **Liquidation IS the expense form** — the exact `AddRequestForPaymentPanel` component, not a lookalike. Only the credit account differs underneath. *(NEU-094)*
6. **Freeze the voucher, not the conversation** — comment thread stays open post-close (marked as post-closure notes). *(NEU-097)*
7. **Multi-booking vouchers (Doctrine D2, `TICKETS.md`)** — one voucher may span many bookings; **booking lives on the line item**. Consequences: per-line booking required for Project/Billable/CA/Reimbursement lines (Office/Direct exempt); routing is **line-aware** with precedence (Forwarding wins → Sir Jayson); expense **rollups read line bookings**; **billable is per line**; voucher-level `booking_id` is only a denormalized "primary booking." Spawned **NEU-107/108/109**; enforcement folds into NEU-088. *(Marcus 7/18)*

## Ground-truth code map (verified via exploration)

**GL / posting**
- `journal_entries` — single table, JSONB `lines[]` (`{account_id, account_code, account_name, debit, credit, description}` + FX), `status: draft|posted|void` (`001_full_schema.sql:835-867`). No line table; positive-only Dr/Cr. `entry_number` auto-assigned by trigger (`091`).
- **All posting = client-side inserts.** No RPC / Edge Function / posting trigger.
- **Balances read only `status='posted'`** (`get_account_balances` RPC mig 230; `AccountLedger.tsx`). `accounts.balance` is dead/static — ignore.
- E-voucher posts at disbursement: `DisburseEVoucherPage.tsx handleConfirm` (posted `JE-DISB`). Invoice/collection write **empty-lines draft** placeholders; real Dr/Cr only via `InvoiceGLPostingSheet.tsx` / `CollectionGLPostingSheet.tsx`. `ModalNewEntry.tsx` is DEAD; live manual path = `NewJournalEntryScreen.tsx`. GJ UI = `GeneralJournal.tsx`.

**E-Voucher form / workflow**
- One shared form: `AddRequestForPaymentPanel.tsx` (`context`/`transactionType` switch labels). Type labels centralized `evoucherTransactionType.ts`; **create-dropdown lists hardcoded 3×** (`AddRequestForPaymentPanel.tsx:1152-1173`, `RoutingRulesTab.tsx:71-75`).
- Line items: `evoucher_line_items` (`031_`), `catalog_item_id` per line, **no booking per line**.
- Status machine `src/types/evoucher.ts:4-31`; gates + handlers `EVoucherWorkflowPanel.tsx`. `handleReject` target: mgr→`draft`, ceo→`pending_manager` (`:442`). Cancel = owner, `draft`/`rejected` only.
- Submit + routing: `useEVoucherSubmit.ts` → `resolveRouting("evoucher", bag)` → materializes `evouchers.pending_approver_department/role` (`buildVoucherInsert`). RLS keys on it (mig 208, 233 WITH CHECK).
- Disbursement: `DisburseEVoucherPage.tsx` (live), `DisbursementSheet.tsx` (DEAD). Advance → status `disbursed` + `cash_receiver`; **"Confirm Receipt" already exists** (`handleConfirmReceipt`, NEU-050). Liquidation: `LiquidationForm.tsx` (dedicated receipt-row form) → `liquidation_submissions`; verify gated to Treasury (`acct_evouchers:disburse`, mig 234).
- Attachments: bucket `attachments`, `crmAttachments.ts`, written to `evouchers.attachments` JSONB. **BUG root cause: `existingAttachments` never merged into save payload — overwrite drops them** (`AddRequestForPaymentPanel.tsx:162-163,504-517`; `useEVoucherSubmit.ts:206`). Also no post-create edit mutation; verify bucket is public.

**Catalog / COA / routing**
- Catalog: `CatalogManagementPage.tsx`; `catalog_items`/`catalog_categories` — **no account FK**. COA: `accounts` (`parent_id` tree), UI `coa/AccountSidePanel.tsx`. `catalog_categories.side` = revenue|expense|both (`033_`).
- Routing: `routing_rules` (`205_`), domain hardcoded `evoucher`, **role/dept only — no user target, no invoice domain**. UI `RoutingRulesTab.tsx`, resolver `resolveRouting.ts`.

**Invoice / collection / AR**
- `invoices` — **no approval workflow** (created `draft`, manual `draft→posted` via `handleFinalize`). `billing_line_items` carry `catalog_item_id` (revenue side). `collections` FK `invoice_id` + `linked_billings[]`. Real AR/cash posting only via the GL Posting Sheets (empty-lines drafts otherwise).
- Permissions: `usePermission().can(module, action)` from `permission_overrides.module_grants` (matrix is king). Keys in `permissionsConfig.ts` + `accessSchema.ts` + `actionApplicability.ts`. `disburse` action is the precedent for a new gate.

### Phase 3 — Transaction Journal engine (needs NEU-091 + Phase 2) — the long pole

**Design (locked):** ONE table + `kind` discriminator, JSONB lines mirroring `journal_entries` 1:1 for a straight-copy post-through, and ONE shared post-through helper that collapses today's 4 scattered posting inserts.

**`transaction_journal_entries`** (new table):
- `id TEXT PK` (`TJ-<kind>-<ts>`), `entry_number` (`TJ-YYYY-NNNN`, trigger cloned from `091`, counter `tj_entry_counter_<year>`).
- `kind` ∈ `expense|advance|invoice|collection|transfer`; `stage` ∈ `pending|awaiting_ack|ready_to_post|posted|void` (the TJ's OWN lifecycle, distinct from GJ `status`).
- `lines JSONB` — **identical shape to `journal_entries.lines`** (incl. per-line FX); `total_debit/credit`; FX header cols with the **same names** as `journal_entries` (082) for verbatim copy.
- source FKs `evoucher_id/invoice_id/collection_id/booking_id` + `transfer_id`; `description/reference/project_number/customer_name`.
- ack: `disburse_to_user_id/acknowledged_at/acknowledged_by`; transfer: `processed_at/processed_by`; `meta JSONB` (kind-specific: butal amounts, from/to accounts, `pending_process`).
- `posted_journal_entry_id → journal_entries(id)` back-link; **partial unique index on it (WHERE NOT NULL)** = hard anti-double-post invariant. Indexes on `(stage, entry_date)`, `kind`, each source FK.

**Post-through helper** `src/utils/accounting/postTransactionJournal.ts` → `postTransactionJournal(tjId, actor)`: guard idempotency (`stage==='posted'`/back-link set → throw; require `ready_to_post`) + balance + non-empty → insert `journal_entries status='posted'` copying lines/totals/FX/source-FK verbatim (trigger assigns `entry_number`) → set TJ `stage='posted' + posted_journal_entry_id` → run source-doc side-effects via a `kind` switch (the logic the old sheets did). **Replaces** the inserts in `DisburseEVoucherPage`, `GLConfirmationSheet`, `InvoiceGLPostingSheet`, `CollectionGLPostingSheet` — those keep their FX/account UI but the terminal action builds/loads a TJ and calls the helper.
