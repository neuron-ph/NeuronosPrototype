## Doctrine

### D1 вЂ” No orphan accounting movements (booking/project linkage)

**Every accounting movement lives on the project в†’ booking chain. The only movement that may float free is a direct/office expense.** (Established by Marcus, 2026-07-10.)

The **booking** is the specific hook; the **project** is the umbrella the booking rolls up into вЂ” so "booking-linked" and "project-linked" are the same requirement stated at two levels (a booking always carries its project).

| Movement | Linkage requirement |
|---|---|
| **Billing** (charge line) | must be on a booking (в†’ project) |
| **Invoice** | must be on a booking (в†’ project) вЂ” never project-level-only |
| **Collection** | must trace to an invoice в†’ booking (в†’ project) |
| **Expense / E-Voucher вЂ” Project Expense, Billable Project Expense, Cash Advance, Reimbursement** | must be on a project/booking |
| **Expense / E-Voucher вЂ” Direct / Office Expense** (`direct_expense`) | **STANDALONE вЂ” the sole carve-out.** No booking, no project required (overhead not tied to a shipment). Ref NEU-048 taxonomy ("Office Expense"). |

**Enforcement shape:** a movement whose linkage is required cannot be created with a null booking. When the underlying charges aren't on a booking yet (raw quotation/project charges, `booking_id = null`), the creating surface **forces a booking** вЂ” auto-pick when the project has exactly one booking, require a choice when it has several, block when it has none (create a booking first) вЂ” and stamps the charges + the movement with that booking on save. Direct/office expense skips the requirement entirely.

**Rollout:** invoices first (NEU-077), then billings + collections.
